# Frontend Stack — Deep-Research Prompt

> Paste the block below into a deep-research AI. It is self-contained (the AI won't see the repo).
> Focus is the **MVP / closed-pilot cut** — the cheapest frontend transformation that makes the app credible for 10–30 paying adult users in the next few weeks. The final-prototype/launch work is explicitly out of scope beyond a short "what to defer" note.
> When results come back, feed them to Claude to turn into a working plan mapped page-by-page onto the existing frontend.

---

**Role:** You are a senior frontend architect and product designer advising a small, technical two-founder startup. Give me a rigorous, decision-oriented recommendation — not a listicle. Prioritize sources from 2024–2026 and flag anything that has changed recently.

**Your single objective.** Recommend the **cheapest, fastest frontend transformation that gets this app to a credible closed pilot of 10–30 paying adult users in the next few weeks.** This is an MVP-scoped question, not a green-field "best stack" question. Optimize relentlessly for shipping speed and low migration cost for a 2-person team. Treat any work that doesn't move the pilot forward as deferrable, and say so explicitly rather than recommending it now. Bias strongly toward incremental, page-by-page migration over any big-bang rewrite. Your default posture should be skepticism toward adopting a new framework/build step unless the pilot genuinely can't ship well without it — if the current build-step-free setup can be made pilot-credible with a shared design-token CSS layer and light JS, that is likely the answer, and you should defend it.

**The product.** "Concourse" is an AI-orchestrated exam-preparation web app for candidates sitting **EPSO exams** (the EU institutions' recruitment competitions — think civil-service cognitive tests: verbal, numerical, abstract reasoning, plus situational/EU-knowledge). The audience is **adult professionals across Europe** (25–45, multilingual, exam-anxious, studying on both laptop and mobile, often in short daily sessions). The flagship feature, "Compass," is an **adaptive practice engine**: it serves cognitively-tagged practice questions, detects a user's weak-pattern "cognitive fingerprint," and generates the next session to target it. The app also has: a landing/marketing page, magic-link auth, an onboarding intake form, a CV/LinkedIn-PDF upload with an AI "fit" read, a competition catalog picker, an adaptive diagnostic mini-test, a study-plan view (master plan + daily tasks), a profile/dashboard ("/me") with progress and insight panels, and an admin health page. This is a **serious-but-encouraging adult education tool**, NOT a gamified kids' app — the design language should read as trustworthy, focused, calm, and credible (closer to Duolingo-for-adults-meets-Linear/Notion than to a flashy game).

**Current technical reality (the starting point).** Backend is **Python 3 / FastAPI + uvicorn**, SQLAlchemy, Supabase Postgres (EU), Anthropic Haiku for LLM calls, deployed on **Railway with auto-deploy on git push**. The frontend today is **7 hand-written single-file HTML pages** — each page has its own inline `<style>` block and vanilla JavaScript, **no build step, no framework, no component reuse, no shared design system.** It works but doesn't scale: no shared components, duplicated CSS, no state management, inconsistent UX across pages. The team is **2 technically-strong founders** (one leads engineering), values shipping speed, and has an existing rule of "no new toolchain unless justified" — but is now explicitly open to reconsidering that for the frontend.

**The core decision I need you to resolve — through an MVP/pilot lens.** For a pilot shipping in the next few weeks, should Concourse:
- **(A)** Stay build-step-free — modern vanilla approach (e.g. Web Components, native ES modules, a lightweight lib like Alpine.js/htmx/Preact-via-CDN, plus a hand-rolled design-token CSS system) served straight from FastAPI static, OR
- **(B)** Adopt a real frontend framework/build (React/Next, SvelteKit, Vue/Nuxt, SolidStart, Astro, etc.) — and if so, which, and how does it coexist with a FastAPI backend on Railway (separate SPA + API? server-rendered? embedded build output? BFF?), and
- **(C)** Any hybrid worth considering (e.g. Astro for marketing pages + islands for the app; htmx for CRUD + a JS island only for the interactive practice engine).

Answer specifically for the pilot horizon: which option gets a credible product in front of 10–30 paying users fastest with least migration risk? If (B) or (C) only pays off *after* the pilot, say so and keep it out of the MVP recommendation.

**Evaluate options against these criteria, weighted for the MVP/pilot goal — the first three dominate the decision, the rest are tie-breakers:** (1) **development velocity & minimal migration cost** for a 2-person team with limited frontend-specialist bandwidth, migrating incrementally (page-by-page, not big-bang) from the current 7-page HTML setup; (2) **deployment simplicity on Railway alongside a Python backend** (avoid ops sprawl — anything that adds a second deploy target or CI complexity is a strike against for the pilot); (3) **"good enough to be credible" design quality for paying adults** — coherent, trustworthy, consistent across pages (not maximum design ceiling — that's post-pilot); (4) **fit for the one genuinely interactive surface, the adaptive quiz UI** (immediate feedback, timers, progress, result screens) — note where the current vanilla approach starts to hurt; (5) **accessibility & mobile-first basics** (adult users on phones, WCAG basics — full i18n/multilingual is likely post-pilot, confirm); (6) **maintainability & AI-assisted-coding friendliness**; (7) **performance** (fast first load matters for the marketing/conversion page).

**Also cover, concretely:**
- The **component/design-system layer** specifically: recommend an approach for design tokens, theming, and a reusable component set. Compare headless/unstyled libraries (Radix, Headless UI, Ark, shadcn/ui, Tailwind, Park UI, DaisyUI, PandaCSS, vanilla-extract, plain CSS custom properties) with a bias toward something a small team can wield fast and that looks intentional, not templated.
- **Charts/data-viz** options for the progress dashboard and cognitive-fingerprint panels (mastery over time, dimensional radar/heatmap), weighted toward light, dependency-lean choices.
- **Auth integration**: the app uses Supabase magic-link auth today via server-set signed cookies — note any framework choice's implications for keeping/replacing that.
- **The adaptive-quiz UX pattern**: reference how the best adaptive-learning / spaced-practice / test-prep products (Duolingo, Khan Academy, Brilliant, Quizlet, Elevate/Lumosity, RemNote/Anki-style, Sana, Synthesis) structure their practice-session UI, feedback loops, and progress/insight surfaces — and what specifically translates to a credible *adult professional* exam-prep context (what to borrow, what to avoid as too childish).
- **AI-in-the-loop UI patterns**: since content is LLM-generated and the engine reasons about the user, note UX patterns for surfacing AI-generated insights trustworthily (loading/streaming states, "why am I seeing this," graceful degradation when the LLM is slow or fails).

**Deliverable format** (the MVP path is the main deliverable — spend most of your effort on items 3–4):
1. **A clear recommendation** for the pilot (pick one primary path, name the runner-up, and state the specific trigger that would make you switch post-pilot).
2. **A comparison table** of the top 3–4 stack options scored against the MVP-weighted criteria above (make the weighting explicit; velocity + migration cost + deploy simplicity should dominate the scores).
3. **A concrete recommended MVP stack** — framework (or none), component/styling layer, charts lib, animation approach, deployment topology on Railway with FastAPI — with a one-sentence justification per choice. Keep it lean: prefer the fewest new moving parts that clear the "credible for paying adults" bar.
4. **A concrete MVP execution plan for the closed pilot** (this is the core ask):
   - The **smallest transformation** that makes the app feel coherent and credible for 10–30 paying adult users in the next few weeks.
   - **What to standardize first** (design tokens, shared shell/nav, typography, buttons/forms) and how to introduce it without a big-bang rewrite.
   - **Which of the ~7 pages matter most** for pilot credibility and conversion, in priority order, and which can stay near-untouched for now.
   - A rough **effort/sequencing estimate** for a 2-person team (order of operations, what's parallelizable).
   - An explicit **"defer to post-pilot" list** so we don't gold-plate: which things to consciously NOT do yet (full component-library maturity, animation polish, dark mode/theming, full i18n, deep accessibility hardening, marketing-page craft) — one line each on why it's safe to defer.
5. **Risks, gotchas, and "do NOT do this" anti-patterns** for a 2-person team on a tight pilot timeline — including the honest case for NOT adopting a framework at all if that's genuinely the right call, and the traps of a mid-migration half-framework state.

Be opinionated and MVP-first. If the current build-step-free approach is the right call for the pilot, say so and defend it. If a framework is clearly worth the migration *before* the pilot, name exactly which one and why, and give the migration a realistic effort estimate for a 2-person team — otherwise, keep framework adoption in the deferred/post-pilot section.
